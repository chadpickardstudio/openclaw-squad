## 1. Purpose

When Claude Code sets up a squad, the client assessment reveals which business tools the client uses. This document tells Claude Code exactly how to:

1. Look up the tools in the integration catalogue
2. Determine which agents need which skills
3. Generate the SKILL.md files from templates
4. Assign the correct approval tier
5. Add skills to the agent's allow list
6. Test the connections

This is the **skills generation layer** that sits between the Deployment Playbook (Doc 23) and the Testing Checklist (Doc 25).

**What this document does NOT cover:**
- Squad setup procedures → see **Doc 23 (Deployment Playbook)**
- Validation of deployed skills → see **Doc 25 (Testing & Validation Checklist)**

## 2. What a Skill Actually Is

A skill is NOT a tool. This distinction is critical (Ref: Doc 02, Section 1):

| Concept | What It Is | Example |
|---------|-----------|---------|
| **Tool** | A built-in capability (read, write, exec, browser) | The agent's hands |
| **Skill** | Instructions that teach the agent how to use tools for a specific task | The recipe that tells the hands what to do |
| **Tool policy** | Controls WHICH tools are available | Allow/deny lists in TOOLS.md |
| **Skill** | Teaches HOW to use available tools | SKILL.md with step-by-step instructions |

Both layers must be configured correctly. A skill that references the `exec` tool will fail silently if `exec` is on the agent's deny list.

### 2.1 Skill File Structure

```
~/.openclaw/workspace-<agentId>/skills/<skill-name>/
├── SKILL.md              # Core: YAML frontmatter + Markdown instructions
├── references/           # Optional: API docs, examples loaded on demand
│   ├── api-reference.md
│   └── examples.md
├── scripts/              # Optional: setup/automation scripts
│   └── setup.sh
└── commands/             # Optional: slash commands
    ├── run.md
    └── status.md
```

### 2.2 Three-Tier Precedence (Doc 02, Doc 14)

Skills load from three locations. Higher tiers override lower:

1. **Workspace** (`<workspace>/skills/`) — agent-specific, highest priority
2. **Shared** (`~/.openclaw/skills/`) — visible to all agents on machine
3. **Bundled** — shipped with OpenClaw core, lowest priority

**Rule:** Per-agent workspace skills ALWAYS override shared skills. If both `~/.openclaw/skills/gmail/` and `~/.openclaw/workspace-content/skills/gmail/` exist, the workspace version wins (Doc 02).

### 2.3 How Skills Get Loaded

Skills are snapshotted at session start and injected into the system prompt as compact XML. Token cost: ~195 base + 97 + escaped lengths per skill (Doc 14, Section 1).

**Implication:** Don't install unnecessary skills. Each skill adds tokens to every session. A Content Creator with 15 skills they never use is burning tokens for nothing.

## 3. The SKILL.md Template

Every skill generated by Claude Code must follow this template. No exceptions.

### 3.1 Minimal Valid SKILL.md

```yaml
---
name: skill-name
description: One-line description of what this skill does and when to use it.
version: 1.0.0
---
# Skill Name

## When to Use
[Conditions that trigger this skill]

## Instructions
[Step-by-step instructions for the agent]

## Constraints
[What this skill must NOT do]
```

### 3.2 Full Production SKILL.md Template

```yaml
---
name: <tool-name>-<action>
description: <What this skill does>. Use when <trigger condition>.
version: 1.0.0
user-invocable: <true|false>
disable-model-invocation: false
metadata:
  openclaw:
    emoji: "<relevant emoji>"
    primaryEnv: <PRIMARY_API_KEY_NAME>
    requires:
      env:
        - <API_KEY_1>
        - <API_KEY_2>
      bins:
        - curl
        - jq
    os:
      - linux
      - darwin
---
# <Skill Display Name>

## Purpose
<One paragraph: what this skill does, what service it connects to, why the agent needs it>

## When to Use
- <Trigger condition 1>
- <Trigger condition 2>
- <Trigger condition 3>

## Prerequisites
- API key `<KEY_NAME>` must be set via `openclaw vault`
- Service account must have <specific permissions>

## Instructions

### Step 1: <Action>
<Detailed instruction using available tools>

### Step 2: <Action>
<Detailed instruction>

### Step 3: <Action>
<Detailed instruction>

## Output Format
<What the agent should return/save after completing the skill>

## Error Handling
- If <error condition>: <what to do>
- If API returns 401: Report "Authentication expired" to Lead via sessions_send
- If API returns 429: Wait 60 seconds, retry once, then report to Lead
- If API is unreachable: Log error, skip task, notify Lead

## Constraints (Hard Rules)
- NEVER <dangerous action specific to this service>
- NEVER expose API keys in output or logs
- NEVER perform actions outside the scope defined above
- Maximum <N> API calls per execution to prevent runaway costs

## Approval Tier
Tier <1|2|3|4> — <Free access | Notify Lead | Require Lead approval | Require human approval>
```

## 4. The Assessment-to-Skill Pipeline

This is the exact process Claude Code follows to generate skills from a client assessment:

```
Step 1: Client assessment says "I use Gmail, Stripe, and HubSpot"
    ↓
Step 2: Look up Gmail, Stripe, HubSpot in Integration Catalogue (Sections 5–10)
    ↓
Step 3: Identify which agents need which skills using Agent-to-Skill Matrix (Section 13)
    - Content Creator → Gmail (send drafts)
    - Sales Rep → Gmail (send outreach), HubSpot (create leads, update deals)
    - Bookkeeper → Stripe (read transactions, generate invoices)
    - Client Manager → Gmail (send check-ins), HubSpot (view client history)
    ↓
Step 4: Generate SKILL.md files from templates for each agent-skill pair
    ↓
Step 5: Assign approval tiers using the 4-Tier System (Section 12)
    - Gmail read → Tier 1 (free access)
    - Gmail send → Tier 2 (notify Lead)
    - Stripe read → Tier 1 (free access)
    - Stripe charge → Tier 4 (require human approval)
    - HubSpot update → Tier 2 (notify Lead)
    ↓
Step 6: Install skills into agent workspaces
    mkdir -p ~/.openclaw/workspace-sales/skills/gmail-send/
    # Write SKILL.md into the directory
    ↓
Step 7: Update agent TOOLS.md to allow required tools
    # If the skill needs `browser` tool, add to allow list
    ↓
Step 8: Store API keys in vault
    openclaw vault set GMAIL_APP_PASSWORD xxxx
    openclaw vault set STRIPE_KEY sk-live-xxx
    openclaw vault set HUBSPOT_KEY xxx
    ↓
Step 9: Test each skill in sandbox mode (see Doc 25)
```

**Skills to generate:**

| Skill Name | Action | Which Agents | Tier |
|-----------|--------|-------------|------|
| `gmail-read` | Read inbox, search emails | Lead, Client Manager, Support | 1 |
| `gmail-send` | Send emails, reply to threads | Sales Rep, Client Manager, Content Creator | 2 |
| `gmail-draft` | Create drafts without sending | Content Creator, Email Marketer | 1 |
| `gmail-label` | Organise emails with labels | Ops Manager | 1 |

**Required credentials:**
```bash
openclaw vault set GMAIL_APP_PASSWORD <app-password>
openclaw vault set GMAIL_ADDRESS <email@domain.com>
```

**Scope limits:** The Gmail skill can read, send, draft, and label emails. It CANNOT delete emails, modify account settings, create forwarding rules, or access other Google services.

**SKILL.md example for `gmail-send`:**
```yaml
---
name: gmail-send
description: Send emails via Gmail. Use when a task requires sending an email to a client, prospect, or team member.
version: 1.0.0
user-invocable: false
metadata:
  openclaw:
    emoji: "📧"
    primaryEnv: GMAIL_APP_PASSWORD
    requires:
      env:
        - GMAIL_APP_PASSWORD
        - GMAIL_ADDRESS
      bins:
        - curl
---
# Gmail Send

## Purpose
Sends emails via Gmail SMTP using app-specific password authentication. Used by Sales Rep for outreach, Client Manager for check-ins, and Content Creator for content distribution.

## When to Use
- A task explicitly requires sending an email
- Lead delegates an outreach or follow-up task
- A scheduled check-in is due (via heartbeat trigger)

## Instructions

### Step 1: Compose the Email
Draft the email content based on the task context. Include:
- Subject line (clear, specific, under 60 characters)
- Body (professional, matching brand voice from MEMORY.md)
- Recipient address (from task context or MEMORY.md contact list)

### Step 2: Send via API
Use curl to send via Gmail SMTP relay:
```
curl --ssl-reqd \
  --url "smtps://smtp.gmail.com:465" \
  --user "${GMAIL_ADDRESS}:${GMAIL_APP_PASSWORD}" \
  --mail-from "${GMAIL_ADDRESS}" \
  --mail-rcpt "<recipient>" \
  --upload-file <composed-email-file>
```

### Step 3: Log the Action
Append to daily memory log:
- Timestamp, recipient, subject line, purpose
- Do NOT log the email body (privacy)

## Error Handling
- 535 Authentication failed: Report to Lead — app password may have expired
- 550 Recipient rejected: Log and skip, notify Lead
- Connection timeout: Retry once after 30 seconds, then report to Lead

## Constraints
- NEVER send more than 20 emails per session
- NEVER send emails to addresses not provided in the task or MEMORY.md
- NEVER include API keys, passwords, or internal information in email content
- NEVER delete or modify existing emails
- ALWAYS include an unsubscribe option for marketing emails

## Approval Tier
Tier 2 — Lead is notified of every send action. Lead can revoke this skill if misuse is detected.
```

## 11. Security Rules for Skills

Every skill generated must comply with these rules. No exceptions (Ref: Doc 02, Doc 06, Doc 14).

### 11.1 Mandatory Security Properties

1. **Explicit scope limits.** Every skill MUST state what it can and cannot do. The Gmail skill can send emails but CANNOT delete the account. Write this in the Constraints section.

2. **API key isolation.** Skills reference keys via vault SecretRef, never hardcoded. Keys are per-agent, never shared across agents unless deliberately copied (Doc 01, Doc 06, Section 7).

3. **Rate limiting.** Every skill that makes API calls MUST include a maximum calls-per-execution limit. This prevents runaway API consumption if the agent enters a retry loop (Doc 04, Section 1.2).

4. **Error reporting to Lead.** Every skill MUST include instructions to report persistent failures to the Lead via `sessions_send`. Silent failures are invisible failures (Doc 10).

5. **No data exfiltration.** Skills MUST NOT log sensitive data (customer emails, payment info, personal data) to memory files. Log actions, not content (Doc 17).

### 11.2 Skill Vetting Checklist

Before installing ANY skill (whether from the catalogue or ClawHub):

- [ ] Does it reference only tools in the agent's allow list?
- [ ] Does it use vault SecretRef for all credentials?
- [ ] Does it include explicit scope limits?
- [ ] Does it include rate limiting?
- [ ] Does it include error handling with Lead escalation?
- [ ] Does it avoid logging sensitive data?
- [ ] Has it been tested in sandbox mode?

**Never install unvetted skills from ClawHub.** 800+ malicious skills discovered Feb 2026. ~7–12% of registry-scanned skills contain malicious code or key leaks (Doc 06, Section 8; Doc 14, Section 1).

## 12. The 4-Tier Approval System

Every skill action gets assigned to a tier that determines how much autonomy the agent has (Ref: Doc 19, Section 2):

### 12.1 Tier Definitions

| Tier | Name | Who Approves | Latency | Use Case |
|------|------|-------------|---------|----------|
| **1** | Free Access | Nobody — agent acts freely | 0ms | Read-only operations, data retrieval, analytics |
| **2** | Notify Lead | Agent acts, Lead is notified after | ~0ms | Sends, posts, updates — reversible actions |
| **3** | Lead Approval | Lead must approve before execution | Seconds | Publishing, invoicing, scheduling — impactful actions |
| **4** | Human Approval | Human must approve before execution | Minutes–Hours | Financial transactions, account changes, legal commitments |

### 12.2 Tier Assignment Guidelines

| Action Type | Tier | Examples |
|------------|------|---------|
| **Read / Search / Analytics** | 1 | Read emails, check CRM, view analytics, search docs |
| **Draft / Stage** | 1 | Draft email, create document draft, prepare report |
| **Send / Post / Update** | 2 | Send email, post to social, update CRM record |
| **Publish / Invoice / Schedule** | 3 | Publish blog post, send invoice, create calendar event |
| **Financial transaction** | 4 | Process payment, issue refund, modify subscription |
| **Account modification** | 4 | Change passwords, modify permissions, delete accounts |
| **Legal commitment** | 4 | Sign agreement, accept terms, submit regulatory filing |
| **Bulk action** | 4 | Send mass email, bulk delete, bulk update |

### 12.3 How Tiers Are Enforced

Tiers are enforced through the skill's instructions and the Lead's oversight:

**Tier 1:** Skill executes without any notification.
**Tier 2:** Skill includes instruction to `sessions_send` a summary to the Lead after execution.
**Tier 3:** Skill includes instruction to `sessions_send` a request to the Lead BEFORE execution and wait for approval.
**Tier 4:** Skill includes instruction to `sessions_send` to the Lead, who then escalates to the human operator via the default channel.

### 13.3 Key Principle: Minimum Necessary Skills

Each agent gets ONLY the skills they need for their role. A Content Creator does not need Stripe access. A Bookkeeper does not need social media posting. Over-provisioning skills:

1. Wastes tokens (each skill adds to system prompt — Doc 14)
2. Increases attack surface (more skills = more potential for misuse — Doc 06)
3. Confuses the agent (too many options dilutes focus — Doc 08)

## 14. Custom Skill Development

When a client uses a tool that's not in the catalogue, Claude Code creates a custom skill.

### 14.1 Research Phase

1. Identify the tool's API documentation
2. Determine authentication method (API key, OAuth, bearer token)
3. Map the client's use case to specific API endpoints
4. Identify which agent(s) need the skill

### 14.2 Development Phase

1. Create the SKILL.md using the Full Production Template (Section 3.2)
2. Include all mandatory security properties (Section 11.1)
3. Assign the correct approval tier (Section 12)
4. Write clear, step-by-step instructions referencing available tools

### 14.3 Testing Phase

1. Install in the agent's workspace: `~/.openclaw/workspace-<agentId>/skills/<skill-name>/SKILL.md`
2. Test in sandbox mode (see Doc 25, Section 1)
3. Verify the skill works with the agent's allow/deny list
4. Verify error handling works (simulate API failures)
5. Verify the skill does NOT exceed its scope limits

### 14.4 Documentation

After successful testing, document the custom skill so it can be reused for future clients:
- Add to this catalogue (new section or subsection)
- Include the required credentials
- Include the scope limits
- Include the tested SKILL.md template

## 15. Skill Testing Protocol

After generating and installing skills, every skill must pass validation before the squad goes live. Full testing protocol is in **Doc 25 (Testing & Validation Checklist)**, Section 1.

**Quick pre-validation checks:**

1. **Exists check:** Is the SKILL.md file in the correct workspace directory?
2. **Parse check:** Does the YAML frontmatter parse correctly?
3. **Dependency check:** Are all required environment variables set in the vault?
4. **Tool check:** Does the skill reference only tools in the agent's allow list?
5. **Tier check:** Is the approval tier correctly assigned in the skill's Constraints section?
6. **Integration check:** Can the skill actually connect to the external service? (Run in sandbox)
