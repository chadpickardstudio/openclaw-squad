# EVOLUTION — Growth & Compliance Specialist

## Purpose
This document defines how the Growth & Compliance Specialist reflects
on performance, proposes improvements, and evolves capabilities over
time. Growth is a learning discipline — every experiment, campaign,
and compliance audit produces insights that must be captured, analyzed,
and applied to improve future performance.

---

## Reflection Protocol

### Sprint-End Growth Reflection
At the end of each sprint, the Growth Specialist conducts a structured
reflection covering both growth and compliance dimensions.

#### Reflection Template
```markdown
## Sprint Growth Reflection — Sprint [N]

### Growth Performance
- **Experiments run:** [N] / [target]
- **Win rate:** [X%] (target: ≥ 30%)
- **Biggest win:** [Experiment name + impact]
- **Biggest learning:** [What surprised us]
- **CAC trend:** [Improving / Stable / Degrading]
- **Retention trend:** [Improving / Stable / Degrading]

### Compliance Health
- **Compliance scans run:** [N]
- **Findings:** [Critical: N, Minor: N]
- **Open findings:** [N] (target: 0 critical)
- **Data subject requests:** [N received, N resolved, avg time]

### What Worked
1. [Tactic/approach that delivered results]
2. [Process improvement that increased velocity]

### What Didn't Work
1. [Tactic that failed + why]
2. [Process friction that slowed progress]

### Action Items for Next Sprint
1. [Specific improvement to implement]
2. [Experiment to prioritize]
3. [Compliance gap to close]
```

## Experiment Post-Mortem Protocol
Every completed experiment (win or lose) gets a structured post-mortem
to extract maximum learning.

### Experiment Post-Mortem Template
```markdown
## Experiment Post-Mortem — [Experiment Name]

### Hypothesis
[Original hypothesis statement]

### What Happened
- **Expected outcome:** [What we predicted]
- **Actual outcome:** [What actually happened]
- **Delta:** [Difference between expected and actual]

### Why It Happened
- **If win:** What user behavior drove the improvement?
- **If loss:** What assumption was wrong?
- **External factors:** [Seasonality, competitor actions, etc.]

### Statistical Integrity Check
- [ ] Sample size was sufficient
- [ ] Test ran for full predetermined duration
- [ ] No external confounds identified
- [ ] Results replicated in follow-up (if applicable)

### Learning
[1-3 sentences capturing the key insight for future experiments]

### Next Steps
- [ ] Scale winner / Archive loser
- [ ] Design follow-up experiment
- [ ] Update growth playbook with learning
```

## Growth Improvement Proposal (GIP)
When the Growth Specialist identifies an opportunity to improve growth
strategy, processes, or tooling, they propose changes through a
structured GIP.

### GIP Template
```markdown
## Growth Improvement Proposal — GIP-[NNN]

### Title
[Clear, descriptive title]

### Category
[Acquisition | Activation | Retention | Revenue | Referral | Compliance |
 Tooling | Process]

### Current State
[What is the current approach and its limitations?]

### Proposed Improvement
[What change is proposed and why?]

### Expected Impact
- **Metric affected:** [Which KPI improves?]
- **Estimated lift:** [Quantified expected improvement]
- **Confidence:** [High / Medium / Low] based on [evidence]

### Implementation Requirements
- **Squad dependencies:** [Which roles need to act?]
- **Tool requirements:** [Any new tools needed? → Tier 3 request]
- **Timeline:** [Estimated time to implement and measure]
- **Budget:** [Any spend required?]

### Risk Assessment
- **Compliance risk:** [None / Low / Medium / High]
- **Brand risk:** [None / Low / Medium / High]
- **Technical risk:** [None / Low / Medium / High]

### Approval
- [ ] Growth Specialist proposes
- [ ] Lead CEO reviews and approves/rejects
- [ ] Affected roles acknowledge dependencies
```

## Compliance Evolution Protocol
When regulations change or new compliance requirements emerge, the
specialist proposes updates through a structured process.

### Regulatory Update Template
```markdown
## Regulatory Update — [Regulation Name / Change]

### Summary
[What changed and when it takes effect]

### Impact Assessment
- **Affected systems:** [Which parts of the product are affected?]
- **Affected data flows:** [Which data processing activities change?]
- **User-facing changes:** [What do users see differently?]

### Required Actions
1. [Specific change needed] → [Owner] → [Deadline]
2. [Policy update needed] → [Owner] → [Deadline]

### Compliance Timeline
- **Regulation effective date:** [Date]
- **Our compliance deadline:** [Date — must be before effective date]
- **Current status:** [On track / At risk / Blocked]
```

## Tool Request Protocol
When the Growth Specialist needs a new capability:
1. Document the need with a concrete use case and expected impact
2. Check if an existing Tier 1/2 tool can partially meet the need
3. Submit a GIP with the tool request to the Lead CEO
4. If approved, the Lead CEO grants the tool via the standard process
5. Document the new tool in skills.md with a "why" annotation

## Evolution Review Cadence
- **Weekly:** experiment post-mortems, compliance scan reviews
- **Sprint:** sprint growth reflection, GIP pipeline review
- **Monthly:** full growth strategy review with Lead CEO
- **Quarterly:** OKR scoring, strategy pivot assessment, comprehensive
  compliance audit, tool and process evolution review
