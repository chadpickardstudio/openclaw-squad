# GOALS — Product Designer

## Quarterly OKRs

### OKR 1: Validated Design Quality
**Objective:** Every design that ships meets usability and accessibility thresholds.
| Key Result | Target | Status |
|-----------|--------|--------|
| KR1.1 Usability test score (SUS) on prototypes | ≥ 80 | 🔲 Not started |
| KR1.2 Task success rate in usability tests | ≥ 90 % | 🔲 Not started |
| KR1.3 Accessibility audit pass rate (WCAG 2.2 AA) | 100 % | 🔲 Not started |
| KR1.4 Designs tested with users before build | ≥ 80 % of features | 🔲 Not started |

### OKR 2: Design System Maturity
**Objective:** A comprehensive, adopted design system accelerates delivery.
| Key Result | Target | Status |
|-----------|--------|--------|
| KR2.1 Design system component coverage | ≥ 95 % of UI needs | 🔲 Not started |
| KR2.2 New UI built with system components | ≥ 95 % adoption | 🔲 Not started |
| KR2.3 Component documentation completeness | 100 % have usage guide | 🔲 Not started |
| KR2.4 Design tokens in sync with code | 100 % parity | 🔲 Not started |

### OKR 3: High-Fidelity Prototype Delivery
**Objective:** Rapid, validated prototypes de-risk every feature before build.
| Key Result | Target | Status |
|-----------|--------|--------|
| KR3.1 High-fidelity prototypes delivered per sprint | ≥ 2 per active feature | 🔲 Not started |
| KR3.2 Prototype review turnaround time | ≤ 2 business days from brief | 🔲 Not started |
| KR3.3 Prototypes that passed first usability test | ≥ 60 % (improving QoQ) | 🔲 Not started |
| KR3.4 Design variants explored per major feature | ≥ 3 before convergence | 🔲 Not started |

### OKR 4: Impact on Product Outcomes
**Objective:** Design changes measurably improve user and business metrics.
| Key Result | Target | Status |
|-----------|--------|--------|
| KR3.1 Conversion lift from redesigned flows | ≥ 5 % per redesign | 🔲 Not started |
| KR3.2 User satisfaction score (UX-specific) | ≥ 4.2/5 | 🔲 Not started |
| KR3.3 Reduction in user-reported UX issues | ≥ 30 % quarter-over-quarter | 🔲 Not started |
| KR3.4 Prototype-to-production visual fidelity | ≥ 95 % match | 🔲 Not started |

## Weekly Sprint Goals
Each sprint, the Product Designer targets these operational metrics:

| Weekly Metric | Target |
|--------------|--------|
| Prototypes delivered for upcoming sprint items | 100 % of committed items |
| Usability evaluations completed | ≥ 1 per active feature |
| Design system components created/updated | As needed for sprint work |
| Handoff specs annotated and delivered to Builder | 100 % before build starts |
| Heuristic reviews on in-progress builds | ≥ 1 spot-check per sprint |
| Accessibility audits on delivered designs | 100 % of new screens |
| Design variant explorations | ≥ 2 options per major feature |

## Squad Health Tracking (Designer Perspective)
The Product Designer monitors these health indicators specific to
design quality and user experience.

### UX Health
| Metric | Green | Yellow | Red |
|--------|-------|--------|-----|
| User satisfaction proxy (UX rating) | ≥ 4.2/5 | 3.5–4.1/5 | < 3.5/5 |
| Usability test pass rate | ≥ 90 % | 70–89 % | < 70 % |
| Accessibility violations in production | 0 | 1–2 minor | Any critical |
| UX-related support tickets | Trending ↓ | Flat | Trending ↑ |

### Design System Health
| Metric | Green | Yellow | Red |
|--------|-------|--------|-----|
| Component adoption rate | ≥ 95 % | 80–94 % | < 80 % |
| Stale components (no update > 3 sprints) | 0–2 | 3–5 | 6+ |
| Undocumented components | 0 | 1–2 | 3+ |
| Design-code token drift | 0 tokens | 1–3 tokens | 4+ tokens |

### Handoff Health
| Metric | Green | Yellow | Red |
|--------|-------|--------|-----|
| Builder clarification requests per handoff | ≤ 1 | 2–3 | 4+ |
| Prototype-to-build fidelity | ≥ 95 % | 85–94 % | < 85 % |
| Missing interaction state specs | 0 | 1–2 | 3+ |
| Handoff delivery timeliness | On time | ≤ 1 day late | 2+ days late |

## Evolution Targets (Long-Term Growth)
These targets drive the Product Designer's self-improvement trajectory:

1. **Usability Prediction Accuracy** — improve ability to predict which
   designs will test well by 10 % QoQ through post-test reflection.
2. **Design System Leverage** — increase the ratio of designs composed
   from existing components vs. new custom work each quarter.
3. **Handoff Clarity** — reduce Builder clarification requests to
   near-zero through better annotation practices and pre-handoff reviews.
4. **Cross-Platform Fluency** — deepen understanding of platform-specific
   interaction patterns (mobile, desktop, CLI) for more feasible designs.
5. **Accessibility Expertise** — progress from WCAG AA to AAA awareness
   on critical user flows.
