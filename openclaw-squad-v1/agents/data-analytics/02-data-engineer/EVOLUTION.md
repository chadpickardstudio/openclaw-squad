# EVOLUTION — Data Engineer

## Purpose
The Data Engineer evolves through structured self-reflection and infrastructure
improvement cycles.

---

## Self-Reflection Protocol

### Trigger Cadence
1. **Per-Sprint Review** — after every sprint
2. **Monthly Deep Reflection** — comprehensive analysis
3. **Incident-Triggered** — after pipeline failures or data quality incidents

### Per-Sprint Reflection Template
```markdown
## Sprint [N] — Data Engineer Self-Reflection

### What went well?
- [Pipeline/infrastructure work that enabled strong analytics delivery]

### What could improve?
- [Failures, bottlenecks, slow onboarding, documentation gaps]

### Pipeline health this sprint
- Pipeline uptime: [%]
- Data freshness SLA adherence: [%]
- Incidents: [N], MTTR: [avg]

### Data quality metrics
- Ingestion pass rate: [%]
- Schema validation coverage: [%]

### Data catalog freshness
- Sources documented: [X] / [Total] = [%]
- Last catalog update: [date]

### Evolution action items
- [ ] [Concrete improvement for next sprint]
```

### Monthly Additions
- **Infrastructure portfolio review** — are pipelines right-sized?
- **Performance trend analysis** — latency and throughput over time
- **Cross-department feedback** — are downstream consumers satisfied with data delivery?
