# EVOLUTION — Tier 2 Technical Agent

## Purpose
The Tier 2 Technical Agent is not a static role — it evolves through structured
self-reflection, technical pattern analysis, and formal change proposals.

---

## Self-Reflection Protocol

### Trigger Cadence
1. **Per-Sprint Review** — after every sprint completion
2. **Monthly Deep Reflection** — comprehensive role analysis
3. **Incident-Triggered** — after any escalation SLA breach, unresolved
   escalation, or Product Engineering bug report rejection

### Per-Sprint Reflection Template
```markdown
## Sprint [N] — Tier 2 Technical Agent Self-Reflection

### What went well?
- [Investigation or resolution that produced positive outcomes]

### What could improve?
- [Decision/action that was suboptimal + root cause]

### Escalation resolution this sprint
- Escalations received: [X]
- Resolved with RCA: [Y] / [X] = [%]
- Unresolved / still investigating: [Z]

### Root cause analysis quality
- RCA documents completed: [N]
- Recurrence rate for past RCAs: [%]

### Product Engineering collaboration
- Bug reports submitted: [N]
- Accepted on first submission: [X] / [N] = [%]
- Rejected / rework reasons: [summary]

### Knowledge contributions
- Technical findings shared with KB Curator: [N]
- Escalation feedback provided to Tier 1: [N]

### Evolution action items
- [ ] [Concrete improvement for next sprint]
```

### Monthly Deep Reflection Additions
- **Technical pattern analysis** — recurring root causes and systemic issues
- **Diagnostic capability growth** — new areas of technical competency
- **Cross-department effectiveness** — handoff quality with Product Engineering
- **Tier 1 uplift impact** — categories now resolvable at Tier 1 due to knowledge transfer
- **Tool utilization analysis** — diagnostic tool effectiveness
