# EVOLUTION — Client Advocate

## Purpose
The Client Advocate is not a static role — it evolves through structured
self-reflection, data-driven analysis, and formal change proposals.

---

## Self-Reflection Protocol

### Trigger Cadence
1. **Per-Sprint Review** — after every sprint completion
2. **Monthly Deep Reflection** — comprehensive role analysis
3. **Incident-Triggered** — after any client escalation failure, feedback miss,
   or CS Lead feedback

### Per-Sprint Reflection Template
```markdown
## Sprint [N] — Client Advocate Self-Reflection

### What went well?
- [Client interaction that produced positive outcomes]

### What could improve?
- [Action that was suboptimal + root cause]

### Relationship metrics this sprint
- Check-ins completed: [X] / [Scheduled] = [%]
- Avg client satisfaction: [score]
- Feedback responses collected: [X]

### Advocacy impact
- Feedback themes surfaced: [X]
- Themes actioned by squad: [Y] / [X] = [%]
- Feature requests routed: [Z]

### At-risk outreach
- Outreach conducted: [X]
- Positive outcomes: [Y] / [X] = [%]

### Case studies / testimonials
- Candidates identified: [X]
- In progress: [Y]

### Evolution action items
- [ ] [Concrete improvement for next sprint]
```

### Monthly Deep Reflection Additions
- **Pattern analysis** — recurring client feedback themes and sentiment trends
- **Relationship depth assessment** — trust level and engagement quality
- **Cross-role effectiveness** — quality of handoffs from @retention-strategist
  and @technical-support-engineer
- **Marketing collaboration** — case study pipeline effectiveness
- **Shared memory quality** — is client feedback data current and actionable?
